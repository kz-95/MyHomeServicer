## Tooling / token efficiency

Use these three to keep context lean and edits precise. Apply every session unless
the user says otherwise.

- **rtk (Rust Token Killer)** — prefix shell commands with `rtk` to compress output
  before it hits context (60–90% savings): `rtk git status`, `rtk npx tsc --noEmit`,
  `rtk grep ...`. Meta: `rtk gain` (savings), `rtk discover` (missed chances),
  `rtk proxy <cmd>` (raw, debugging). Full reference: `~/.claude/RTK.md`.
- **caveman** — ultra-compressed reply mode (~75% fewer tokens). Terse: drop
  articles/filler/pleasantries, fragments OK, all technical substance stays. Code,
  commits, PRs, security warnings stay normal prose. Off: "stop caveman" / "normal
  mode". Levels: `/caveman lite|full|ultra`.
- **serena** — use serena MCP semantic tools for code nav and edits instead of
  whole-file reads. Locate: `find_symbol`, `get_symbols_overview`,
  `find_referencing_symbols`. Edit: `replace_symbol_body`, `insert_after_symbol`,
  `insert_before_symbol`. Call `initial_instructions` once before a coding task.
  Symbol-level beats full-file reads on this codebase (49 models, 1396-line schema).

## Coding rules

- TypeScript everywhere — no plain `.js` files in `backend/src/`
- All money fields use `Decimal` — never `float`/`number` for currency
- All UUIDs generated by Postgres (`gen_random_uuid()`) — not in application code
- Never pass `req.body` directly to Prisma — always pick fields explicitly
- BullMQ job payloads must be validated with Zod before processing
- Source of truth: `schema.prisma` → `schema-notes.md` → `api-doc.md` → `security-notes.md`
- Database workflow is **Prisma migrations** (committed under `backend/prisma/migrations/`). Dev: `npm run db:migrate` (`prisma migrate dev`) to create + apply a migration. Prod (Railway): `prisma migrate deploy` runs automatically in the start command. `npm run db:reset` = `prisma migrate reset --force` (drops, re-applies all migrations, reseeds). `db:reset-test` resets + `seed:test`.
- **After ANY schema change:** stop the running server (frees the Windows DLL lock — `query_engine-windows.dll.node` makes `prisma generate` fail silently while the server runs, leaving a stale client that throws P2022 on login), then run `npm run db:migrate` (`prisma migrate dev --name <change>`) to generate + apply the migration and regenerate the client, commit the new folder under `prisma/migrations/`, and restart. Never edit an already-applied migration — add a new one.
- Run `npx tsc --noEmit` after every edit to `backend/src/` (zero errors = gate)
- Run `npx tsc --noEmit` in `frontend/` after completion
- **After `npm install <package>` in backend or frontend:** always commit the updated `package-lock.json` in the same commit as the code change. The bat launchers (`Run.bat`, `Run-Clean.bat`, `Run-Test.bat`) detect a newer `package-lock.json` and auto-run `npm install` — no manual bat edits needed. Committing `package-lock.json` is what triggers this on other machines.
- **CI pipeline** — three event-driven workflows (see `docs/superpowers/specs/2026-06-10-ci-pipeline-design.md`):
  - `push-checks.yml` — lint + build + unit tests on every push (~3 min)
  - `pr-gate.yml` — full suite (lint, build, unit, API E2E, browser E2E, secret scan, npm audit) on PR to master (~10 min). This is the main quality gate.
  - `nightly.yml` — maintenance sweep (E2E, secret scan, npm audit) at 2am MYT (~6 min)
  - All three include a WhatsApp notification job via Meta Cloud API (free: 1000 msgs/month). Set `META_TOKEN`, `META_PHONE_ID`, `META_WHATSAPP_TO` GitHub secrets to enable. Skipped silently if not set.
  - `security.yml` is deleted — folded into `pr-gate.yml` + `nightly.yml`. Old `ci.yml` replaced by this design.
  - Never add `on: push` to the full E2E suite or secret scan — use the PR gate and nightly schedule as designed.

## Documentation sync rule

Every change must update the relevant Markdown docs in the **same session** as
the code change:

- A feature add/change/removal updates `TODO.md`
- A schema change updates `docs/ai-context/schema-notes.md`
- An endpoint change updates `docs/api-reference/api-doc.md`
- Every agent logs its work to its own `*-log.md`

`TODO.md` is the single source of truth for task state. If code and docs
disagree, the docs are corrected **before** the task is considered done.

---

## Task completion protocol

When a task is finished, run these steps in order **every time**:

1. **Update relevant Markdown docs** in the same session (per the Documentation
   sync rule above) — `TODO.md`, schema/API/log docs as applicable.
2. **Summarize in chat** — short summary of what changed and why.
3. **Commit + push** — stage the task's changes, write a clear commit message,
   then push.
   - **Default: push to `master`** when working solo / sequentially.
   - **Parallel multi-role work:** each role (backend, frontend, qa, devops)
     pushes to its own branch (e.g. `backend`, `frontend`, `qa`, `devops`) to
     avoid clobbering, then those merge to `master`.
   - Commit only the files this task touched; do not blanket `git add -A` over
     unrelated working-tree changes.
   - End commit messages with the Co-Authored-By trailer.
   - Never `--no-verify`; if a hook fails, fix the cause.

A task is "finished" only after all three steps complete. Use `rtk git ...` for
the git commands to keep output compact.
