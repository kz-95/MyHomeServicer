### BUG-001 (original report) - `slotEndTime` uses server local time, not Malaysia time (UTC+8)

- **Found by:** QA agent (2026-05-21)
- **File:** `backend/src/services/booking.service.ts`, function `slotEndTime` (line ~19)
- **Severity:** High - affects production deployments on UTC servers (standard Docker/cloud default)
- **Symptom:** `slotEndTime(date, 'morning')` calls `end.setHours(12, 0, 0, 0)`, which uses the **server's local timezone**. On a UTC server the "morning" slot ends at `12:00 UTC = 20:00 MYT` - **8 hours late**. The `noshow.detect` BullMQ job is then scheduled 30 min after this window, so it fires ~8 hours too late (or too early for other timezone offsets).
- **Repro:**
  ```ts
  // On a UTC server:
  const date = new Date('2026-06-15T00:00:00.000Z');
  slotEndTime(date, 'morning').toISOString(); // → "2026-06-15T12:00:00.000Z"
  // Expected: "2026-06-15T04:00:00.000Z"  (12:00 MYT = 04:00 UTC)
  ```
- **Fix (Backend):** Replace `setHours` with an explicit UTC+8 offset:
  ```ts
  export function slotEndTime(date: Date, slot: TimeSlot): Date {
    const end = new Date(date);
    const MYT_OFFSET_MS = 8 * 60 * 60_000;
    const slotEndHourUTC = SLOT_END_HOUR[slot] - 8; // e.g. morning: 12 - 8 = 4
    end.setUTCHours(slotEndHourUTC < 0 ? slotEndHourUTC + 24 : slotEndHourUTC, 0, 0, 0);
    if (slotEndHourUTC < 0) end.setUTCDate(end.getUTCDate() - 1);
    return end;
  }
  ```
  Or, simpler: set `TZ=Asia/Kuala_Lumpur` in the server environment (Docker / systemd / `.env`). The latter is the lowest-risk change for V1.
- **Workaround until fixed:** Set `TZ=Asia/Kuala_Lumpur` in `docker-compose.yml` and `backend/.env`. This is the recommended V1 fix - a single env var, no code change, no risk of regression in other date logic.
- **✅ DevOps action completed (2026-05-21):** `TZ=Asia/Kuala_Lumpur` added to `backend/.env.example` and to `docker-compose.yml` (`backend` + `worker` services, `full` profile). Now defence-in-depth alongside the Backend code fix above.

### BUG-005 - Socket.io CORS only accepted `APP_URL`; rejected LAN/tunnel WebSocket handshakes - ✅ FIXED

**Backend fix applied 2026-05-23.**

- **Found by:** Backend agent (2026-05-23)
- **File:** `backend/src/socket/index.ts`, `initSocket` function
- **Severity:** Low - dev-only, no production impact (production uses a single origin)
- **Symptom:** `initSocket` passed a single string (`env.APP_URL`) to Socket.io's `cors.origin`.
  The HTTP CORS middleware in `app.ts` accepted a *set* of origins (`APP_URL + CORS_EXTRA_ORIGINS`).
  When a developer set `CORS_EXTRA_ORIGINS` for LAN/tunnel testing, HTTP requests passed but
  WebSocket upgrade handshakes were rejected with a CORS error - Socket.io events never fired.
- **Fix:** Compute `corsOrigins` using the same pattern as `app.ts` and pass the array to Socket.io:
  ```ts
  const corsOrigins = [env.APP_URL, ...env.CORS_EXTRA_ORIGINS.split(',').map((o) => o.trim())].filter(Boolean);
  io = new IOServer(server, { cors: { origin: corsOrigins, credentials: true } });
  ```
- **No API contract change. No schema change. No other agent action required.**

### Infrastructure finding: mask.test.ts and http.test.ts swallowed by Jest worker crash

**Symptom:** `npx jest --listTests` lists 14 files. `npm test` main run output shows only 12 -
`mask.test.ts` and `http.test.ts` produce no output, no pass/fail, no suite summary.

**Root cause:**

1. Jest runs suites in parallel across worker processes.
2. `auth-lockout.test.ts` imports `bcrypt` - compiled for Windows. On Linux: `invalid ELF header`
   crashes the entire Node worker process immediately.
3. `mask.test.ts` and `http.test.ts` are co-scheduled in the same crashing worker.
4. When a Jest worker process crashes, all results for its scheduled batch are silently dropped.
5. Running `npx jest tests/unit/mask.test.ts tests/unit/http.test.ts` alone avoids the crashing
   worker and both suites pass cleanly.

**Classification:** Linux sandbox limitation only. Not a code regression. On Windows (`bcrypt`
and Prisma binaries correct) all 14 suites run in a single `npm test` pass.

**Resolution:** No code change needed. Workaround: run the two suites individually in the
sandbox when confirmation is needed.
